Chapter 01

React Router v6

Navigasi dan routing di aplikasi React
1.1

Setup & BrowserRouter

Cara kerja React Router dan titik awal konfigurasi
Apa itu

React Router adalah library yang memungkinkan kamu membuat navigasi antar halaman di aplikasi React tanpa melakukan full page reload. URL berubah, konten berganti, tapi browser tidak reload.

Kenapa penting

Setiap aplikasi multi-halaman membutuhkan routing. Ini adalah skill wajib yang selalu ditanyakan di interview. Tanpa React Router, aplikasimu hanya satu halaman statis.

Kapan dipakai
  • Saat aplikasi punya lebih dari satu halaman
  • Saat perlu navigasi login → dashboard
  • Saat URL harus mencerminkan konten yang ditampilkan
Mental Model

Bayangkan BrowserRouter sebagai "GPS" aplikasimu. Dia memantau URL browser dan memberitahu React komponen mana yang harus ditampilkan berdasarkan URL tersebut.

bash instalasi 
npm install react-router-dom
jsx src/main.jsx 
import { BrowserRouter } from 'react-router-dom';
import App from './App';

ReactDOM.createRoot(document.getElementById('root')).render(
  <BrowserRouter>
    <App />
  </BrowserRouter>
);
TaskFlow - Langkah 1

Kita bungkus seluruh aplikasi TaskFlow dengan BrowserRouter di main.jsx. Ini adalah satu-satunya tempat kita meletakkan BrowserRouter, yaitu paling atas di tree komponen.

1.2

Routes, Route, dan URL Parameters

Mendefinisikan halaman dan membaca data dari URL
Apa itu

Routes adalah wadah untuk semua route. Route adalah aturan "jika URL ini, tampilkan komponen ini". URL params (:id) memungkinkan kamu menyematkan data dinamis di URL.

Kenapa penting

Hampir setiap halaman detail menggunakan URL params, contohnya /tasks/42. Memahami cara membaca params dari URL adalah skill dasar yang wajib dikuasai.

Kapan dipakai
  • Halaman detail produk: /produk/:id
  • Profil user: /user/:username
  • Halaman edit: /tasks/:id/edit
jsx src/router/AppRouter.jsx 
import { Routes, Route } from 'react-router-dom';
import LoginPage from '../pages/LoginPage';
import DashboardPage from '../pages/DashboardPage';
import TaskDetailPage from '../pages/TaskDetailPage';

export default function AppRouter() {
  return (
    <Routes>
      <Route path="/"          element={<LoginPage />}          />
      <Route path="/dashboard" element={<DashboardPage />}      />
      <Route path="/tasks/:id" element={<TaskDetailPage />}     />
      <Route path="*"          element={<div>404 - Halaman tidak ditemukan</div>} />
    </Routes>
  );
}
jsx src/pages/TaskDetailPage.jsx - membaca URL param 
import { useParams } from 'react-router-dom';

export default function TaskDetailPage() {
  const { id } = useParams(); // membaca :id dari URL /tasks/42 => id = "42"

  return (
    <div>
      <h1>Detail Task #{id}</h1>
      // fetch task berdasarkan id (akan diintegrasikan di Chapter 2)
    </div>
  );
}
jsx navigasi menggunakan Link - BUKAN tag <a> 
import { Link, NavLink } from 'react-router-dom';

function Navbar() {
  return (
    <nav>
      // Link = navigasi tanpa reload, NavLink = Link dengan class "active" otomatis
      <NavLink
        to="/dashboard"
        className={({ isActive }) => isActive ? 'nav-link active' : 'nav-link'}
      >
        Dashboard
      </NavLink>

      // Link ke halaman task dengan ID dinamis
      <Link to={`/tasks/${task.id}`}>Lihat Detail</Link>
    </nav>
  );
}
Kesalahan Umum
  • Menggunakan <a href="/dashboard"> untuk navigasi internal - ini akan menyebabkan full page reload dan kehilangan semua state. Selalu gunakan <Link>
  • Lupa bahwa nilai dari useParams() selalu bertipe string, bukan number. Pastikan konversi dengan Number(id) atau parseInt(id) saat perlu
  • Meletakkan <Routes> di luar <BrowserRouter> - ini akan error
1.3

Navigasi Programatik dengan useNavigate

Berpindah halaman dari dalam kode JavaScript
Apa itu

useNavigate adalah hook yang mengembalikan fungsi navigate. Fungsi ini digunakan untuk berpindah halaman dari dalam logika JavaScript, bukan dari klik user secara langsung.

Kenapa penting

Kamu tidak bisa selalu menggunakan <Link>. Setelah submit form login, setelah delete data, atau setelah aksi async selesai, kamu perlu redirect secara programatik.

Kapan dipakai
  • Setelah login berhasil → redirect ke dashboard
  • Setelah submit form → redirect ke halaman list
  • Tombol "kembali" kustom
jsx src/pages/LoginPage.jsx 
import { useNavigate } from 'react-router-dom';

export default function LoginPage() {
  const navigate = useNavigate();

  const handleLogin = async (data) => {
    try {
      await loginApi(data);             // panggil API login
      navigate('/dashboard');          // redirect setelah berhasil
    } catch (err) {
      console.error('Login gagal', err);
    }
  };

  return <button onClick={handleLogin}>Login</button>;
}

// navigate(-1)         => kembali ke halaman sebelumnya (seperti tombol back)
// navigate('/login', { replace: true })  => replace history, tidak bisa back
Kesalahan Umum
  • Memanggil navigate() langsung di body komponen (bukan di dalam event handler atau useEffect) - ini menyebabkan infinite render
  • Lupa replace: true setelah logout sehingga user bisa menekan "back" dan kembali ke halaman yang butuh autentikasi
1.4

Protected Routes (Guard)

Mencegah akses ke halaman tanpa login
Apa itu

Protected Route adalah pola di mana kamu membungkus Route dengan komponen yang memeriksa status autentikasi. Jika user belum login, dia diarahkan ke halaman login secara otomatis.

Kenapa penting

Ini adalah pola yang wajib ada di hampir setiap aplikasi production. Tanpa ini, siapapun bisa mengakses halaman dashboard atau admin hanya dengan mengetik URL secara langsung.

Kapan dipakai
  • Halaman dashboard yang butuh login
  • Halaman admin yang butuh role tertentu
  • Semua route yang butuh autentikasi
jsx src/router/ProtectedRoute.jsx 
import { Navigate } from 'react-router-dom';
import { useAuthStore } from '../stores/useAuthStore'; // Zustand store (CH 5)

export default function ProtectedRoute({ children }) {
  const isLoggedIn = useAuthStore((state) => state.isLoggedIn);

  if (!isLoggedIn) {
    // replace: true agar user tidak bisa "back" ke halaman protected
    return <Navigate to="/" replace />;
  }

  return children;
}
jsx src/router/AppRouter.jsx - menggunakan ProtectedRoute 
import ProtectedRoute from './ProtectedRoute';

export default function AppRouter() {
  return (
    <Routes>
      <Route path="/" element={<LoginPage />} />

      // Semua route di bawah ini butuh login
      <Route
        path="/dashboard"
        element={
          <ProtectedRoute>
            <DashboardPage />
          </ProtectedRoute>
        }
      />
      <Route
        path="/tasks/:id"
        element={
          <ProtectedRoute>
            <TaskDetailPage />
          </ProtectedRoute>
        }
      />
    </Routes>
  );
}
Kesalahan Umum
  • Mengecek autentikasi hanya dari localStorage tanpa state global - state tidak reaktif, tampilan tidak update saat logout
  • Tidak menggunakan replace pada Navigate setelah logout - user bisa kembali ke halaman protected dengan tombol back browser
1.5

Nested Routes & Layout Route

Berbagi layout (Navbar, Sidebar) antar halaman
Apa itu

Nested Routes memungkinkan route bersarang di dalam route lain. Layout Route digunakan untuk berbagi komponen layout (seperti Navbar) tanpa mengulang kode di setiap halaman.

Kenapa penting

Ini adalah pola yang sangat umum di aplikasi production. Navbar dan Sidebar yang konsisten di semua halaman dashboard diimplementasikan dengan nested routes, bukan dengan mengcopy komponen ke setiap halaman.

Kapan dipakai
  • Semua halaman dashboard berbagi satu layout
  • Tab navigasi di dalam satu halaman
  • Halaman admin dengan sidebar khusus
jsx src/components/layout/AppLayout.jsx - layout bersama 
import { Outlet } from 'react-router-dom';
import Navbar from './Navbar';
import Sidebar from './Sidebar';

// Outlet = placeholder untuk child route yang aktif
export default function AppLayout() {
  return (
    <div>
      <Navbar />
      <div style={{ display: 'flex' }}>
        <Sidebar />
        <main>
          <Outlet /> // Di sini child route dirender
        </main>
      </div>
    </div>
  );
}
jsx src/router/AppRouter.jsx - nested routes final TaskFlow 
export default function AppRouter() {
  return (
    <Routes>
      // Route publik
      <Route path="/" element={<LoginPage />} />

      // Layout route - semua child akan mewarisi AppLayout
      <Route
        element={
          <ProtectedRoute>
            <AppLayout />
          </ProtectedRoute>
        }
      >
        <Route path="/dashboard"  element={<DashboardPage />}  />
        <Route path="/tasks/:id" element={<TaskDetailPage />} />
      </Route>
    </Routes>
  );
}

Pola ini sangat umum di dunia kerja. Hampir semua aplikasi production menggunakan layout route untuk Navbar dan Sidebar. Kuasai pola ini dengan baik sebelum interview.

Kesalahan Umum
  • Lupa menambahkan <Outlet /> di komponen layout - child route tidak akan pernah ditampilkan
  • Copy-paste Navbar ke setiap halaman secara manual alih-alih menggunakan layout route
Chapter 02

TanStack Query + Axios

Server state management dan HTTP client
2.1

Setup QueryClient & Instance Axios

Konfigurasi awal yang diperlukan sebelum fetch data
Apa itu

TanStack Query (dulu React Query) adalah library untuk mengurus semua hal yang berhubungan dengan data dari server: fetching, caching, loading state, error state, dan sinkronisasi otomatis. Axios adalah HTTP client yang lebih powerful dari fetch bawaan browser.

Kenapa penting

Mengurus server state dengan useState + useEffect itu menyakitkan: kamu harus handle loading, error, refetch, dan cache secara manual. TanStack Query melakukan semua itu secara otomatis dengan kode yang jauh lebih bersih.

Kapan dipakai
  • Setiap kali fetch data dari API REST
  • Saat butuh caching otomatis
  • Saat butuh refetch saat window kembali fokus
  • Saat melakukan create/update/delete via API
Mental Model

Bayangkan TanStack Query sebagai "asisten cerdas" untuk data API kamu. Kamu cukup bilang "ambilkan data tasks", dan dia yang mengurus loading state, error handling, caching, dan pembaruan otomatis. Kamu tidak perlu lagi menulis setLoading(true) dan setData(result) secara manual.

bash instalasi 
npm install @tanstack/react-query axios
jsx src/main.jsx - setup QueryClientProvider 
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { BrowserRouter } from 'react-router-dom';

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 5, // data dianggap fresh selama 5 menit
      retry: 1,               // coba ulang 1x jika gagal
    },
  },
});

ReactDOM.createRoot(document.getElementById('root')).render(
  <QueryClientProvider client={queryClient}>
    <BrowserRouter>
      <App />
    </BrowserRouter>
  </QueryClientProvider>
);
jsx src/api/axiosInstance.js - konfigurasi Axios 
import axios from 'axios';

const api = axios.create({
  baseURL: 'https://api.taskflow.dev/v1', // base URL sekali, tidak perlu diulang
  timeout: 10000,                           // batas waktu 10 detik
  headers: {
    'Content-Type': 'application/json',
  },
});

export default api;
2.2

useQuery - Mengambil Data dari API

Fetch data dengan loading, error, dan caching otomatis
Apa itu

useQuery adalah hook untuk mengambil (GET) data dari server. Dia secara otomatis mengurus loading state, error state, caching, dan refetch.

Kenapa penting

Menggantikan pola useEffect + fetch + useState yang verbose dan error-prone. Dengan useQuery, kode fetch data menjadi 3-5 baris dan lebih reliable.

Kapan dipakai
  • Fetch daftar tasks di DashboardPage
  • Fetch detail task berdasarkan ID
  • Fetch data apapun dari API GET
jsx src/api/taskApi.js - fungsi API 
import api from './axiosInstance';

// Fungsi-fungsi ini hanya bertanggung jawab untuk panggil API
export const fetchTasks = () => api.get('/tasks').then((res) => res.data);
export const fetchTaskById = (id) => api.get(`/tasks/${id}`).then((res) => res.data);
jsx src/pages/DashboardPage.jsx - useQuery di action 
import { useQuery } from '@tanstack/react-query';
import { fetchTasks } from '../api/taskApi';

export default function DashboardPage() {
  const {
    data: tasks,       // data yang dikembalikan dari queryFn
    isLoading,         // true saat pertama kali loading
    isError,           // true jika request gagal
    error,             // objek error jika isError = true
    refetch,           // fungsi untuk fetch ulang manual
  } = useQuery({
    queryKey: ['tasks'],    // key unik untuk cache - PENTING!
    queryFn: fetchTasks,    // fungsi yang melakukan fetch
  });

  if (isLoading) return <div>Memuat tasks...</div>;
  if (isError) return <div>Error: {error.message}</div>;

  return (
    <div>
      {tasks.map((task) => (
        <TaskCard key={task.id} task={task} />
      ))}
    </div>
  );
}
jsx src/pages/TaskDetailPage.jsx - useQuery dengan ID dinamis 
import { useParams } from 'react-router-dom';
import { useQuery } from '@tanstack/react-query';
import { fetchTaskById } from '../api/taskApi';

export default function TaskDetailPage() {
  const { id } = useParams();

  const { data: task, isLoading } = useQuery({
    queryKey: ['tasks', id],           // key mengandung id agar cache terpisah per task
    queryFn: () => fetchTaskById(id),  // fungsi dengan parameter
    enabled: !!id,                       // hanya fetch jika id ada
  });

  if (isLoading) return <div>Memuat...</div>;
  return <div>{task?.title}</div>;
}
Kesalahan Umum
  • Menggunakan queryKey: ['tasks'] yang sama untuk query dengan ID berbeda - semua task akan berbagi cache yang sama. Selalu sertakan ID di queryKey: ['tasks', id]
  • Langsung menaruh fetch() di dalam queryFn tanpa fungsi terpisah - susah di-test dan duplikasi kode
  • Tidak menangani isLoading dan isError - menyebabkan error runtime ketika data belum tersedia
2.3

useMutation - Create, Update, Delete Data

Mengirim data ke server dengan feedback otomatis
Apa itu

useMutation adalah hook untuk operasi yang mengubah data di server: POST, PUT, PATCH, DELETE. Berbeda dari useQuery, mutation hanya dijalankan ketika kamu memintanya secara eksplisit.

Kenapa penting

Setiap aplikasi CRUD butuh mutation. useMutation menyediakan onSuccess, onError, dan isPending secara gratis, sehingga kamu bisa memberikan feedback yang baik ke user.

Kapan dipakai
  • Membuat task baru (POST)
  • Mengupdate status task (PATCH)
  • Menghapus task (DELETE)
  • Semua operasi yang mengubah data server
jsx src/api/taskApi.js - tambahkan fungsi mutasi 
export const createTask = (data) => api.post('/tasks', data).then((res) => res.data);
export const updateTask = ({ id, ...data }) => api.patch(`/tasks/${id}`, data).then((res) => res.data);
export const deleteTask = (id) => api.delete(`/tasks/${id}`).then((res) => res.data);
jsx src/pages/DashboardPage.jsx - useMutation untuk hapus task 
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { deleteTask } from '../api/taskApi';

function TaskCard({ task }) {
  const queryClient = useQueryClient(); // akses ke query cache

  const deleteMutation = useMutation({
    mutationFn: deleteTask,
    onSuccess: () => {
      // Setelah hapus berhasil, invalidate cache tasks agar list direfresh
      queryClient.invalidateQueries({ queryKey: ['tasks'] });
    },
    onError: (error) => {
      console.error('Gagal menghapus task:', error);
    },
  });

  return (
    <div>
      <p>{task.title}</p>
      <button
        onClick={() => deleteMutation.mutate(task.id)}
        disabled={deleteMutation.isPending}  // disable saat loading
      >
        {deleteMutation.isPending ? 'Menghapus...' : 'Hapus'}
      </button>
    </div>
  );
}
Kesalahan Umum
  • Tidak memanggil invalidateQueries setelah mutation berhasil - list data di UI tidak akan diperbarui setelah create/update/delete
  • Menggunakan mutate() tanpa menangani isPending - user bisa klik tombol berkali-kali dan mengirim request duplikat
2.4

Cache & Query Invalidation

Cara TanStack Query mengingat data dan kapan harus memperbarui
Mental Model

Cache = lemari penyimpanan data sementara. queryKey = label di lemari itu. invalidateQueries = kamu memberi tanda "sudah kedaluwarsa" pada label tertentu sehingga TanStack Query akan mengambil data segar dari server di request berikutnya.

jsx konsep cache - queryKey sebagai kunci unik 
// Setiap queryKey yang unik punya cache sendiri
useQuery({ queryKey: ['tasks'] })             // cache: semua tasks
useQuery({ queryKey: ['tasks', '1'] })        // cache: task dengan id 1
useQuery({ queryKey: ['tasks', '2'] })        // cache: task dengan id 2
useQuery({ queryKey: ['tasks', { status: 'done' }] }) // cache: tasks dengan filter

// Invalidate berdasarkan prefix key
queryClient.invalidateQueries({ queryKey: ['tasks'] });
// Ini akan menginvalidasi SEMUA query yang keynya dimulai dengan 'tasks'
// Termasuk ['tasks'], ['tasks', '1'], ['tasks', '2'], dll.
jsx staleTime vs gcTime - perbedaan penting 
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 5,  // 5 menit: data dianggap "fresh", tidak refetch
      gcTime: 1000 * 60 * 10,   // 10 menit: data dihapus dari cache setelah ini
    },
  },
});

// staleTime = 0 (default): setiap mount komponen akan refetch
// staleTime = Infinity: data tidak pernah dianggap stale (jarang dipakai)
// Untuk data yang jarang berubah (daftar kategori), staleTime tinggi lebih efisien
2.5

Axios Interceptor untuk Auth Token

Menambahkan token JWT otomatis di setiap request
Apa itu

Interceptor adalah middleware di Axios. Request interceptor berjalan sebelum request dikirim, response interceptor berjalan setelah response diterima. Keduanya memungkinkan kamu mengubah atau menangani data secara terpusat.

Kenapa penting

Tanpa interceptor, kamu harus menambahkan token auth secara manual di setiap request. Dengan interceptor, cukup satu kali konfigurasi dan semua request otomatis menyertakan token.

Kapan dipakai
  • Menambahkan JWT token ke header Authorization
  • Handle error 401 (unauthorized) - redirect ke login
  • Logging semua request/response
jsx src/api/axiosInstance.js - interceptor lengkap 
import axios from 'axios';

const api = axios.create({
  baseURL: 'https://api.taskflow.dev/v1',
  timeout: 10000,
});

// Request interceptor: tambahkan token sebelum request dikirim
api.interceptors.request.use(
  (config) => {
    const token = localStorage.getItem('token');
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    return config; // harus return config!
  },
  (error) => Promise.reject(error)
);

// Response interceptor: handle error 401 secara global
api.interceptors.response.use(
  (response) => response, // jika sukses, lewatkan saja
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token');
      window.location.href = '/'; // redirect ke login
    }
    return Promise.reject(error);
  }
);

export default api;
Kesalahan Umum
  • Lupa return config di dalam request interceptor - semua request akan ter-hang dan tidak pernah dikirim
  • Membuat banyak instance Axios terpisah alih-alih menggunakan satu instance yang dibagikan

3.1

Setup & Hook useForm

Dasar React Hook Form dan cara mendaftarkan input
Apa itu

React Hook Form (RHF) adalah library form yang performant karena tidak menyimpan nilai setiap input ke React state (tidak ada re-render di setiap ketikan). Zod adalah library untuk membuat schema validasi dengan TypeScript.

Kenapa penting

Form dengan validasi adalah fitur yang ada di hampir setiap aplikasi. Membuat form dengan useState biasa menjadi sangat verbose. RHF + Zod adalah kombinasi yang paling banyak dipakai di industri saat ini.

Kapan dipakai
  • Form login dan registrasi
  • Form buat atau edit task di TaskFlow
  • Form yang butuh validasi kompleks
  • Form dengan banyak field
Mental Model

RHF menggunakan "uncontrolled inputs" - dia mengakses nilai input langsung dari DOM, bukan melalui React state. Ini membuat form sangat cepat. register() adalah cara RHF "mendaftarkan" input agar bisa diawasi.

bash instalasi 
npm install react-hook-form zod @hookform/resolvers
jsx src/components/TaskForm.jsx - useForm dasar 
import { useForm } from 'react-hook-form';

export default function TaskForm({ onSubmit }) {
  const {
    register,       // mendaftarkan input ke RHF
    handleSubmit,   // wrapper submit yang menangani validasi
    formState: { errors, isSubmitting }, // state form
    reset,          // reset form ke nilai awal
  } = useForm();

  const submitHandler = (data) => {
    // data = { title: '...', description: '...', priority: '...' }
    onSubmit(data);
    reset();
  };

  return (
    <form onSubmit={handleSubmit(submitHandler)}>
      <input
        placeholder="Judul task"
        {...register('title')}  // spread operator untuk register
      />
      <button type="submit" disabled={isSubmitting}>
        Simpan
      </button>
    </form>
  );
}
Kesalahan Umum
  • Menggunakan onSubmit={submitHandler} langsung di form tanpa membungkusnya dengan handleSubmit() - validasi tidak akan berjalan
  • Menambahkan value dan onChange secara manual pada input yang sudah di-register - ini konflik dan menyebabkan bug
3.2

Validasi Form dengan Zod Schema

Mendefinisikan aturan validasi dengan cara yang reusable
Apa itu

Zod memungkinkan kamu mendefinisikan "schema" - sebuah objek yang mendeskripsikan bentuk dan aturan validasi data yang valid. Schema ini kemudian dihubungkan ke RHF melalui zodResolver.

Kenapa penting

Validasi form yang baik adalah kualitas yang membedakan developer junior yang baik. Zod membuat validasi mudah dibaca, mudah diubah, dan bisa digunakan ulang di backend dan frontend.

Kapan dipakai
  • Setiap form yang butuh validasi
  • Validasi: required, min/max, format email, regex
  • Validasi silang antar field (misal: konfirmasi password)
jsx src/components/TaskForm.jsx - RHF + Zod lengkap 
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { z } from 'zod';
import { useMutation, useQueryClient } from '@tanstack/react-query';
import { createTask } from '../api/taskApi';

// 1. Definisikan schema validasi
const taskSchema = z.object({
  title: z.string()
    .min(3, 'Judul minimal 3 karakter')
    .max(100, 'Judul maksimal 100 karakter'),
  description: z.string().optional(),
  priority: z.enum(['low', 'medium', 'high'], {
    errorMap: () => ({ message: 'Pilih prioritas yang valid' }),
  }),
  dueDate: z.string().refine(
    (val) => !isNaN(new Date(val)),
    { message: 'Format tanggal tidak valid' }
  ),
});

// Ekstrak tipe TypeScript dari schema (bonus: type safety)
type TaskFormData = z.infer<typeof taskSchema>;

export default function TaskForm({ onClose }) {
  const queryClient = useQueryClient();

  const { register, handleSubmit, formState: { errors } } = useForm({
    resolver: zodResolver(taskSchema), // 2. Hubungkan schema ke RHF
    defaultValues: {
      priority: 'medium',
    },
  });

  const mutation = useMutation({
    mutationFn: createTask,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['tasks'] });
      onClose(); // tutup modal setelah berhasil
    },
  });

  return (
    <form onSubmit={handleSubmit((data) => mutation.mutate(data))}>

      <div>
        <input placeholder="Judul task" {...register('title')} />
        {errors.title && <p className="error">{errors.title.message}</p>}
      </div>

      <div>
        <select {...register('priority')}>
          <option value="low">Rendah</option>
          <option value="medium">Sedang</option>
          <option value="high">Tinggi</option>
        </select>
        {errors.priority && <p className="error">{errors.priority.message}</p>}
      </div>

      <button type="submit" disabled={mutation.isPending}>
        {mutation.isPending ? 'Menyimpan...' : 'Buat Task'}
      </button>
    </form>
  );
}

Error Handling & Edit Form

Menampilkan error validasi dan mengisi ulang form untuk edit
jsx form edit task - prefill defaultValues dari data API 
// Untuk form edit, kita perlu mengisi ulang field dengan data yang ada
function EditTaskForm({ taskId }) {
  const { data: task } = useQuery({
    queryKey: ['tasks', taskId],
    queryFn: () => fetchTaskById(taskId),
  });

  const { register, handleSubmit, formState: { errors }, reset } = useForm({
    resolver: zodResolver(taskSchema),
  });

  // Ketika data task tersedia, isi ulang form
  useEffect(() => {
    if (task) {
      reset({
        title: task.title,
        description: task.description,
        priority: task.priority,
      });
    }
  }, [task, reset]);

  const updateMutation = useMutation({
    mutationFn: (data) => updateTask({ id: taskId, ...data }),
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ['tasks'] }),
  });

  return (
    <form onSubmit={handleSubmit((data) => updateMutation.mutate(data))}>
      <div>
        <label>Judul</label>
        <input {...register('title')} />
        // Tampilkan pesan error spesifik dari Zod
        {errors.title && (
          <span style={{ color: 'red', fontSize: '12px' }}>
            {errors.title.message}
          </span>
        )}
      </div>
      <button type="submit">Update</button>
    </form>
  );
}
Kesalahan Umum
  • Mengisi defaultValues langsung dari data yang bisa undefined - menyebabkan form tidak ter-populate. Selalu gunakan reset() di dalam useEffect setelah data tersedia
  • Lupa menambahkan reset ke dalam dependency array useEffect sehingga ESLint terus warning
  • Menggunakan watch() di setiap field untuk tracking nilai - ini menyebabkan re-render berlebihan. Hanya gunakan watch jika benar-benar perlu
Chapter 04

Redux Toolkit (RTK)

State management global untuk aplikasi skala besar
4.1

Setup Store dan Provider

Konfigurasi Redux Toolkit dari nol
Apa itu

Redux Toolkit (RTK) adalah cara modern dan resmi untuk menggunakan Redux. Redux adalah library state management global berbasis "single source of truth": semua state aplikasi disimpan di satu tempat bernama "Store".

Kenapa penting

Redux masih digunakan di banyak perusahaan dan sering ditanyakan di interview. RTK menghilangkan boilerplate Redux lama yang sangat verbose, sehingga lebih praktis dan modern.

Kapan dipakai
  • State UI global: sidebar open/close, notifikasi toast
  • Aplikasi besar dengan banyak fitur yang berbagi state
  • Tim besar yang butuh standar dan predictability
Mental Model Redux

Store = satu objek besar yang menyimpan semua state. Slice = satu bagian/fitur dari store (contoh: uiSlice, authSlice). Action = pesan "terjadi sesuatu" yang dikirim ke store. Reducer = fungsi yang memutuskan bagaimana state berubah berdasarkan action.

bash instalasi 
npm install @reduxjs/toolkit react-redux
jsx src/store/store.js - membuat store 
import { configureStore } from '@reduxjs/toolkit';
import uiReducer from '../features/ui/uiSlice';

export const store = configureStore({
  reducer: {
    ui: uiReducer,      // state.ui -> semua state UI
    // auth: authReducer, // jika menggunakan Redux untuk auth
  },
});
jsx src/main.jsx - tambahkan Provider 
import { Provider } from 'react-redux';
import { store } from './store/store';

ReactDOM.createRoot(document.getElementById('root')).render(
  <Provider store={store}>
    <QueryClientProvider client={queryClient}>
      <BrowserRouter>
        <App />
      </BrowserRouter>
    </QueryClientProvider>
  </Provider>
);
4.2

createSlice - Mendefinisikan State dan Reducer

Cara modern mendefinisikan potongan state dan logika perubahannya
Apa itu

createSlice adalah fungsi RTK yang menggabungkan state awal, reducer functions, dan action creators dalam satu objek. Ini menghilangkan boilerplate Redux lama yang membutuhkan file terpisah untuk actions dan reducers.

Kenapa penting

Ini adalah cara utama mendefinisikan state di RTK. Semua logika perubahan state untuk satu fitur diletakkan di satu tempat (slice), membuat kode mudah dibaca dan dirawat.

Kapan dipakai
  • Mendefinisikan state global untuk satu fitur
  • Sidebar open/close, modal open/close
  • Notifikasi toast global
jsx src/features/ui/uiSlice.js - UI state untuk TaskFlow 
import { createSlice } from '@reduxjs/toolkit';

const uiSlice = createSlice({
  name: 'ui',                      // nama slice (dipakai sebagai prefix action type)
  initialState: {
    isSidebarOpen: true,
    isCreateTaskModalOpen: false,
    toast: null,               // { message: '...', type: 'success' | 'error' }
  },
  reducers: {
    // Setiap fungsi di sini otomatis menjadi action creator
    toggleSidebar(state) {
      state.isSidebarOpen = !state.isSidebarOpen; // Immer: boleh mutasi langsung!
    },
    openCreateTaskModal(state) {
      state.isCreateTaskModalOpen = true;
    },
    closeCreateTaskModal(state) {
      state.isCreateTaskModalOpen = false;
    },
    showToast(state, action) {      // action.payload = data yang dikirim
      state.toast = action.payload;  // { message: '...', type: 'success' }
    },
    hideToast(state) {
      state.toast = null;
    },
  },
});

// Export action creators
export const {
  toggleSidebar,
  openCreateTaskModal,
  closeCreateTaskModal,
  showToast,
  hideToast,
} = uiSlice.actions;

// Export reducer untuk dimasukkan ke store
export default uiSlice.reducer;

Immer di balik layar: RTK menggunakan library Immer, sehingga di dalam reducer kamu boleh "memutasi" state secara langsung seperti state.isSidebarOpen = true. Ini tidak benar-benar memutasi state asli, Immer yang membuatnya immutable di balik layar.

4.3

createAsyncThunk untuk Operasi Async

Menangani logika async (API call) di dalam Redux
Apa itu

createAsyncThunk adalah cara Redux menangani operasi asynchronous seperti API call. Dia secara otomatis men-dispatch tiga action: pending, fulfilled, dan rejected.

Kenapa penting

Sering muncul di interview dan di codebase lama yang menggunakan Redux untuk server state. Penting untuk dipahami meskipun di project baru kamu akan lebih sering menggunakan TanStack Query untuk ini.

Kapan dipakai
  • Ketika proyek sudah pakai Redux tapi belum ada TanStack Query
  • Proses async yang bukan data fetching murni (contoh: login yang menyimpan token ke state)

Penting untuk interview: Di proyek modern, createAsyncThunk tidak direkomendasikan untuk data fetching. Gunakan TanStack Query untuk itu. Namun kamu harus paham cara kerjanya karena sering ada di kodebase yang sudah ada.

jsx contoh createAsyncThunk untuk login (dipahami, bukan dipakai di TaskFlow) 
import { createAsyncThunk, createSlice } from '@reduxjs/toolkit';
import api from '../api/axiosInstance';

// createAsyncThunk(actionName, asyncFunction)
export const loginUser = createAsyncThunk(
  'auth/loginUser',
  async (credentials, thunkAPI) => {
    try {
      const response = await api.post('/auth/login', credentials);
      return response.data; // menjadi action.payload di fulfilled
    } catch (error) {
      return thunkAPI.rejectWithValue(error.response?.data?.message);
    }
  }
);

const authSlice = createSlice({
  name: 'auth',
  initialState: { user: null, status: 'idle', error: null },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(loginUser.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(loginUser.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.user = action.payload.user;
      })
      .addCase(loginUser.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.payload;
      });
  },
});
4.4

useSelector & useDispatch di Komponen

Cara membaca state dan mengirim aksi dari komponen React
Apa itu

useSelector membaca data dari Redux store. useDispatch mengembalikan fungsi dispatch yang digunakan untuk mengirim action ke store.

Kenapa penting

Ini adalah dua hook utama yang dipakai di setiap komponen yang berinteraksi dengan Redux. Tanpa ini, kamu tidak bisa membaca atau mengubah state Redux dari komponen manapun.

Kapan dipakai
  • Setiap komponen yang butuh data dari Redux store
  • Setiap komponen yang perlu mengubah state Redux
jsx src/components/layout/Sidebar.jsx - membaca dan mengubah UI state 
import { useSelector, useDispatch } from 'react-redux';
import { toggleSidebar } from '../../features/ui/uiSlice';

export default function Sidebar() {
  const dispatch = useDispatch();

  // useSelector mengambil slice state yang kita butuhkan
  const isSidebarOpen = useSelector((state) => state.ui.isSidebarOpen);

  return (
    <aside style={{ width: isSidebarOpen ? '240px' : '60px' }}>
      <button onClick={() => dispatch(toggleSidebar())}>
        {isSidebarOpen ? 'Tutup' : 'Buka'}
      </button>
      // ... isi sidebar ...
    </aside>
  );
}
jsx src/components/layout/Navbar.jsx - trigger modal buat task 
import { useDispatch, useSelector } from 'react-redux';
import { openCreateTaskModal, closeCreateTaskModal } from '../../features/ui/uiSlice';
import TaskForm from '../TaskForm';

export default function Navbar() {
  const dispatch = useDispatch();
  const isModalOpen = useSelector((state) => state.ui.isCreateTaskModalOpen);

  return (
    <nav>
      <button onClick={() => dispatch(openCreateTaskModal())}>
        + Buat Task
      </button>

      {isModalOpen && (
        <Modal>
          <TaskForm onClose={() => dispatch(closeCreateTaskModal())} />
        </Modal>
      )}
    </nav>
  );
}
Kesalahan Umum
  • Selector yang terlalu lebar: useSelector(state => state.ui) - ini menyebabkan komponen re-render setiap kali state UI berubah meskipun bagian yang dipakai tidak berubah. Selalu ambil data yang spesifik: state => state.ui.isSidebarOpen
  • Menggunakan Redux untuk menyimpan data dari API (server state) - gunakan TanStack Query untuk itu, Redux untuk UI state
  • Memanggil dispatch secara langsung di render (bukan di event handler atau useEffect)
Chapter 05

Zustand

State management global yang minimalis dan tanpa boilerplate
5.1

Setup dan Membuat Store

Cara membuat state global dengan Zustand dalam hitungan menit
Apa itu

Zustand adalah library state management yang sangat ringan dan simpel. Berbeda dari Redux, tidak ada boilerplate, tidak ada Provider, tidak ada action type, tidak ada reducer. Cukup create() dan selesai.

Kenapa penting

Semakin populer di tahun-tahun terakhir sebagai alternatif Redux yang lebih sederhana. Banyak startup dan tim kecil memilih Zustand karena kemudahannya. Sangat sering muncul di lowongan pekerjaan frontend modern.

Kapan dipakai
  • State auth: user yang sedang login, token
  • Proyek kecil-menengah yang butuh state global
  • Ketika Redux terasa terlalu berat untuk kebutuhan
Mental Model

Zustand store adalah sebuah hook biasa. Kamu membuat useAuthStore dan memanggilnya di komponen manapun, persis seperti hook React biasa. Tidak perlu Provider, tidak perlu Provider, tidak ada setup tambahan.

bash instalasi 
npm install zustand
jsx src/stores/useAuthStore.js - auth store untuk TaskFlow 
import { create } from 'zustand';

export const useAuthStore = create((set) => ({
  // State awal
  user: null,
  token: localStorage.getItem('token') || null,
  isLoggedIn: !!localStorage.getItem('token'),

  // Actions: fungsi untuk mengubah state
  login: (user, token) => {
    localStorage.setItem('token', token);
    set({ user, token, isLoggedIn: true });
  },

  logout: () => {
    localStorage.removeItem('token');
    set({ user: null, token: null, isLoggedIn: false });
  },

  updateUser: (userData) => {
    set((state) => ({ user: { ...state.user, ...userData } }));
  },
}));

Menggunakan Store di Komponen

Membaca state dan memanggil action dari Zustand store
jsx src/pages/LoginPage.jsx - integrasi Zustand + RHF + Router 
import { useNavigate } from 'react-router-dom';
import { useAuthStore } from '../stores/useAuthStore';
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { z } from 'zod';
import api from '../api/axiosInstance';

const loginSchema = z.object({
  email: z.string().email('Format email tidak valid'),
  password: z.string().min(6, 'Password minimal 6 karakter'),
});

export default function LoginPage() {
  const navigate = useNavigate();
  const login = useAuthStore((state) => state.login); // ambil hanya fungsi login

  const { register, handleSubmit, formState: { errors, isSubmitting } } = useForm({
    resolver: zodResolver(loginSchema),
  });

  const onSubmit = async (data) => {
    const response = await api.post('/auth/login', data);
    login(response.data.user, response.data.token); // simpan ke Zustand
    navigate('/dashboard');
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input type="email" {...register('email')} placeholder="Email" />
      {errors.email && <p>{errors.email.message}</p>}

      <input type="password" {...register('password')} placeholder="Password" />
      {errors.password && <p>{errors.password.message}</p>}

      <button type="submit" disabled={isSubmitting}>
        {isSubmitting ? 'Masuk...' : 'Masuk'}
      </button>
    </form>
  );
}
jsx src/components/layout/Navbar.jsx - tombol logout 
import { useNavigate } from 'react-router-dom';
import { useAuthStore } from '../../stores/useAuthStore';

export default function Navbar() {
  const navigate = useNavigate();
  const { user, logout } = useAuthStore(); // bisa ambil beberapa sekaligus

  const handleLogout = () => {
    logout();
    navigate('/', { replace: true });
  };

  return (
    <nav>
      <span>Halo, {user?.name}</span>
      <button onClick={handleLogout}>Keluar</button>
    </nav>
  );
}
Kesalahan Umum
  • Mengambil seluruh store: useAuthStore() tanpa selector - komponen akan re-render setiap kali state apapun di store berubah. Gunakan selector: useAuthStore(state => state.user)
  • Menyimpan data sensitif seperti password di Zustand atau localStorage - hanya simpan token
5.3

Zustand vs Redux Toolkit - Kapan Pilih Mana?

Panduan memilih state management yang tepat di interview
Aturan Praktis untuk Interview

Pertanyaan "Redux atau Zustand?" sering muncul di interview. Jawaban yang baik bukan "Zustand lebih bagus" tapi tahu kapan masing-masing tepat digunakan.

Kriteria Zustand Redux Toolkit
Boilerplate Sangat minimal Lebih banyak (slice, store, provider)
Kurva belajar Sangat mudah Lebih curam
DevTools Ada (lebih terbatas) Redux DevTools yang kuat
Skala proyek Kecil hingga menengah Menengah hingga besar
Time-travel debug Tidak ada Ada
Middleware Tersedia tapi manual Built-in dan terstandar
Kapan pilih Startup, proyek baru, tim kecil Enterprise, tim besar, standarisasi

Di TaskFlow: Kita menggunakan Zustand untuk state auth (ringan, tidak butuh DevTools kompleks) dan Redux Toolkit untuk UI state global (bisa berkembang, punya DevTools yang baik). Di dunia nyata, sebuah proyek biasanya hanya menggunakan salah satunya, bukan keduanya.

Apa itu

React Testing Library (RTL) adalah library testing yang mendorong kamu test komponen seperti yang user lakukan (mencari teks, klik tombol), bukan implementasi internal. Vitest adalah test runner modern yang cepat, cocok dipakai bersama Vite.

Render dan Mencari Elemen

Cara merender komponen dan menemukan elemen di DOM virtual
Apa itu

  • Hindari getByTestId kecuali tidak ada pilihan lain

import TaskCard from '../components/TaskCard';

const mockTask = {
  id: 1,
  title: 'Buat fitur login',
  priority: 'high',
  status: 'in-progress',
};

describe('TaskCard', () => {
  it('menampilkan judul task dengan benar', () => {
    render(<TaskCard task={mockTask} />);

    // getByText: cari elemen yang mengandung teks ini
    expect(screen.getByText('Buat fitur login')).toBeInTheDocument();
  });

  it('menampilkan badge prioritas tinggi', () => {
    render(<TaskCard task={mockTask} />);

    // getByRole: cara terbaik untuk mencari elemen berdasarkan aksesibilitas
    const badge = screen.getByRole('status');
    expect(badge).toHaveTextContent('Tinggi');
  });

  it('tidak menampilkan tombol edit ketika isOwner = false', () => {
    render(<TaskCard task={mockTask} isOwner={false} />);

    // queryByRole: tidak throw error jika tidak ditemukan (berbeda dari getByRole)
    expect(screen.queryByRole('button', { name: /edit/i })).not.toBeInTheDocument();
  });
});
jsx prioritas query RTL - dari terbaik ke terburuk 
// Prioritas tinggi ke rendah (gunakan dari atas ke bawah):

screen.getByRole('button', { name: /submit/i })   // TERBAIK - berbasis aksesibilitas
screen.getByLabelText('Email')                     // untuk input dengan label
screen.getByPlaceholderText('Masukkan email')       // kurang ideal tapi OK
screen.getByText('Selamat Datang')                  // untuk konten teks biasa
screen.getByDisplayValue('nilai input')             // untuk input yang sudah terisi
screen.getByAltText('logo')                        // untuk gambar
screen.getByTestId('my-element')                   // TERBURUK - hanya jika tidak ada pilihan

// Perbedaan getBy vs queryBy vs findBy:
// getBy    - throw error jika tidak ditemukan (untuk elemen yang HARUS ada)
// queryBy  - return null jika tidak ditemukan (untuk test "tidak ada")
// findBy   - async, return promise (untuk elemen yang muncul setelah loading)
6.3

User Events dan Async Testing

Mensimulasikan interaksi user dan menunggu operasi async
Apa itu

userEvent dari @testing-library/user-event mensimulasikan interaksi user yang lebih realistis dibanding fireEvent. waitFor dan findBy* digunakan untuk menunggu perubahan DOM yang async.

Kenapa penting

Form, tombol, dan elemen interaktif lainnya adalah bagian yang paling sering perlu ditest. userEvent lebih realistis karena mensimulasikan events yang sebenarnya terjadi saat user berinteraksi.

Kapan dipakai
  • Test submit form dan validasi error
  • Test klik tombol yang mengubah UI
  • Test input teks dan perubahan state
jsx src/__tests__/TaskForm.test.jsx - test form dan validasi 
import { render, screen, waitFor } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import TaskForm from '../components/TaskForm';

describe('TaskForm', () => {
  it('menampilkan error validasi ketika judul kurang dari 3 karakter', async () => {
    const user = userEvent.setup(); // buat instance user event
    render(<TaskForm onClose={vi.fn()} />);

    // 1. Ketik judul yang terlalu pendek
    await user.type(screen.getByPlaceholderText('Judul task'), 'ab');

    // 2. Klik tombol submit
    await user.click(screen.getByRole('button', { name: /buat task/i }));

    // 3. Tunggu dan cek apakah pesan error muncul
    await waitFor(() => {
      expect(screen.getByText('Judul minimal 3 karakter')).toBeInTheDocument();
    });
  });

  it('memanggil onClose setelah form berhasil disubmit', async () => {
    const user = userEvent.setup();
    const mockOnClose = vi.fn(); // mock function untuk cek apakah dipanggil

    render(<TaskForm onClose={mockOnClose} />);

    await user.type(screen.getByPlaceholderText('Judul task'), 'Task baru saya');
    await user.click(screen.getByRole('button', { name: /buat task/i }));

    await waitFor(() => {
      expect(mockOnClose).toHaveBeenCalledOnce();
    });
  });
});
Kesalahan Umum
  • Menggunakan fireEvent.click() alih-alih userEvent.click() - fireEvent lebih rendah level dan tidak mensimulasikan urutan event yang sebenarnya (mousedown, mouseup, click)
  • Lupa await sebelum userEvent.type() dan userEvent.click() - user-event v14 semuanya async
  • Menggunakan getBy untuk elemen yang muncul secara async - gunakan findBy yang mengembalikan promise dan menunggu elemen muncul
6.4

Mocking API dengan vi.mock

Isolasi komponen dari API call nyata saat testing
Apa itu

Mocking adalah teknik menggantikan dependency nyata (seperti API call) dengan versi palsu yang bisa kita kontrol hasilnya. Ini memastikan test tidak bergantung pada network atau server eksternal.

Kenapa penting

Test yang memanggil API nyata itu lambat, tidak reliable, dan bisa gagal karena masalah network. Mock membuat test cepat, terisolasi, dan deterministik - hasilnya selalu sama setiap kali dijalankan.

Kapan dipakai
  • Test komponen yang menggunakan TanStack Query
  • Test komponen yang langsung memanggil API
  • Simulasi kondisi error dari API
jsx src/__tests__/DashboardPage.test.jsx - mock API dengan QueryClientProvider 
import { render, screen } from '@testing-library/react';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { BrowserRouter } from 'react-router-dom';
import DashboardPage from '../pages/DashboardPage';
import * as taskApi from '../api/taskApi';

// Mock seluruh module taskApi
vi.mock('../api/taskApi');

// Helper untuk membungkus komponen dengan semua Provider yang diperlukan
const renderWithProviders = (component) => {
  const queryClient = new QueryClient({
    defaultOptions: { queries: { retry: false } }, // tidak retry di test
  });
  return render(
    <QueryClientProvider client={queryClient}>
      <BrowserRouter>{component}</BrowserRouter>
    </QueryClientProvider>
  );
};

describe('DashboardPage', () => {
  it('menampilkan daftar task setelah data berhasil di-fetch', async () => {
    // Atur apa yang dikembalikan mock
    taskApi.fetchTasks.mockResolvedValue([
      { id: 1, title: 'Task Pertama', status: 'todo' },
      { id: 2, title: 'Task Kedua', status: 'done' },
    ]);

    renderWithProviders(<DashboardPage />);

    // findByText = async, menunggu elemen muncul setelah loading selesai
    expect(await screen.findByText('Task Pertama')).toBeInTheDocument();
    expect(await screen.findByText('Task Kedua')).toBeInTheDocument();
  });

  it('menampilkan pesan error ketika API gagal', async () => {
    // Simulasi API error
    taskApi.fetchTasks.mockRejectedValue(new Error('Network Error'));

    renderWithProviders(<DashboardPage />);

    expect(await screen.findByText(/error/i)).toBeInTheDocument();
  });
});

Buat helper renderWithProviders: Di proyek nyata, buat satu fungsi helper yang membungkus semua Provider (QueryClientProvider, Provider Redux, BrowserRouter) agar tidak mengulang kode di setiap file test. Tempatkan di src/__tests__/utils.tsx.

Kesalahan Umum
  • Tidak membungkus komponen dengan Provider yang diperlukan (QueryClientProvider, Redux Provider) saat rendering di test - ini akan menyebabkan error karena hooks tidak bisa menemukan context
  • Menggunakan getByText untuk konten yang muncul secara async (setelah loading) - gunakan await findByText() untuk menunggu konten muncul
  • Tidak mereset mock antar test dengan vi.clearAllMocks() di afterEach - sisa mock dari test sebelumnya bisa mempengaruhi test berikutnya
